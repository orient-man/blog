## Context

Post summaries on the main listing page are currently generated by `generateExcerpt()` in `src/lib/utils.ts`.
This function applies regex-based stripping to remove all Markdown syntax, producing a plain-text string truncated at 500 characters.
The PostCard component renders this string inside a `<p>` tag with no formatting.

The full post page uses a completely separate rendering path: `evaluate()` from `@mdx-js/mdx` with `remarkGfm`, `rehypePrettyCode`, and `rehypeCopyButton` plugins, rendered inside a `prose`-styled container.

The gap means summaries lose all inline formatting (bold, links, emphasis), blockquote structure, and paragraph separation.

## Goals / Non-Goals

**Goals:**

- Render post summaries with preserved inline Markdown formatting (bold, italic, strikethrough, links, inline code).
- Preserve blockquote styling in summaries (critical for quote-format posts).
- Replace code fences with a clean `...` placeholder.
- Use zero new dependencies — rely entirely on transitive deps already in `node_modules`.
- Keep the existing plain-text excerpt available for SEO meta tags and RSS feeds.

**Non-Goals:**

- Syntax highlighting in excerpts — code blocks are replaced, not rendered.
- Image rendering in excerpts — images are stripped.
- Full MDX component support in excerpts — JSX components are stripped.
- Changing the RSS feed or SEO meta description format.

## Decisions

### D1: Unified/remark pipeline over regex transformation

**Choice:** Build excerpt HTML using `unified` + `remark-parse` + `remark-gfm` + `remark-rehype` + `hast-util-to-html`.

**Alternatives considered:**
- *Enhanced regex:* Could try to selectively preserve some markdown syntax and wrap in HTML tags. Fragile, error-prone, hard to extend. Rejected.
- *Full MDX evaluate():* Would give identical rendering to full posts but is async, heavier, and would render code blocks with syntax highlighting (unwanted in excerpts). Rejected.
- *Third-party markdown-to-HTML library (marked, markdown-it):* Would work but adds a new dependency when we already have the unified ecosystem installed. Rejected per Simplicity principle.

**Rationale:** The unified pipeline is already in `node_modules` as a transitive dependency of `@mdx-js/mdx`.
It gives us AST-level control to selectively keep/strip/replace node types.
It runs synchronously (with `processSync`) and produces a clean HTML string.

### D2: Custom remark plugin for excerpt extraction

**Choice:** A single remark plugin (`remarkExcerptTransform`) that operates on the mdast AST.

**How it works:**
1. Walk top-level `root.children` nodes.
2. For each node, apply rules:
   - `paragraph`, `blockquote`, `list` → keep (count text length toward ~500 char limit).
   - `code` (fenced) → replace with a paragraph containing `...`. Collapse consecutive placeholders.
   - `heading`, `image`, `html`, `mdxJsxFlowElement`, `thematicBreak` → strip (remove from tree).
3. Stop collecting when accumulated plain-text length >= 500 characters.
4. Truncate the last kept node at a word boundary if it pushes over the limit, appending `...`.
5. Remove all nodes after the cutoff from `root.children`.

**Rationale:** Operating at the AST level is robust — no regex edge cases, handles nested structures (e.g., bold inside a blockquote), and is straightforward to extend.

### D3: Separate `htmlExcerpt` field, not replacing `excerpt`

**Choice:** Add `htmlExcerpt: string | undefined` to the `Post` type alongside the existing `excerpt`.

**Alternatives considered:**
- *Replace `excerpt` entirely with HTML:* Would require updating RSS feed generation and SEO meta tags to strip HTML. Increases blast radius unnecessarily.
- *Single field with a format flag:* Over-engineered for two consumers.

**Rationale:** SEO meta descriptions and RSS `<description>` fields require plain text.
Keeping both fields means each consumer uses the appropriate format with no conversion.

### D4: `dangerouslySetInnerHTML` for PostCard rendering

**Choice:** Render `htmlExcerpt` via `dangerouslySetInnerHTML` inside a `<div>` with `prose prose-sm` classes.

**Rationale:** The HTML is generated at build time from our own trusted Markdown content, not from user input.
There is no XSS vector.
This is the standard React pattern for pre-rendered HTML and is simpler than hydrating MDX components client-side.

### D5: Frontmatter stripping before pipeline

**Choice:** Strip the YAML frontmatter (`---...---`) from the raw content string before passing to the remark pipeline.

**Rationale:** `remark-parse` in plain markdown mode treats `---` as a thematic break, not frontmatter.
The `gray-matter` library already strips frontmatter when parsing post metadata, but `post.content` in the current codebase is the raw file content.
Pre-stripping avoids the frontmatter being parsed as markdown content.

## Risks / Trade-offs

- **[`line-clamp` on HTML content]** CSS `line-clamp` works on block containers but may clip mid-element (e.g., cut a blockquote in half visually).
  This is acceptable — the clamp is a visual affordance, not a content boundary.
  The user clicks through to the full post for the complete text.

- **[Transitive dependency stability]** Relying on transitive deps (`unified`, `remark-parse`, etc.) means a major `@mdx-js/mdx` update could theoretically drop them.
  Mitigation: These are core unified ecosystem packages unlikely to be dropped. If needed, they can be added as direct dependencies later.

- **[Build time]** Running a remark pipeline per post adds processing time at build.
  Mitigation: The pipeline is lightweight (no Shiki/syntax highlighting). With ~33 posts, the impact is negligible.
